# Tools

Every script/binary this project provides, in one place, each linking to
the lifecycle doc that explains it in full context. For what these tools
touch (env vars, volumes, ports), see [Reference](4.1-configuration.md).

| Tool | Purpose | Explained in |
|---|---|---|
| `scripts/setup-dev-environment.sh` | Bootstrap a fresh dev machine (Rust toolchain, wasm target, `dioxus-cli`/`wasm-bindgen-cli`, `sccache`, Docker) | [Setup](3.1-setup.md#development-environment-setup) |
| `scripts/services.sh` | Start/stop/restart/status/logs for the local backend + web dev server | [Development](3.2-development.md#quick-start-scripts) |
| `scripts/desktop.sh` | Launch a desktop client against the local backend | [Development](3.2-development.md#quick-start-scripts) |
| `scripts/admin.sh` | Operator CLI (`tile-lite-elite-admin`) wrapper for a **local** dev server — list/delete users, reset passwords, list/delete/force-end games | [Production Support & Maintenance](3.5-production-support.md#admin-cli) |
| `tile-lite-elite-admin` (binary) | The same operator CLI, run directly — locally via `admin.sh`/`cargo run -p admin-cli`, or against a container via `docker compose exec` | [Production Support & Maintenance](3.5-production-support.md#admin-cli) |
| `scripts/deploy-staging.sh` | Build and run the staging stack locally: `up`/`down`/`reset`/`at <ref>`/`at prod`/`verify` | [Testing & Staging](3.3-testing-and-staging.md#staging-environment) |
| `scripts/deploy.sh` | Build images locally and ship them to the production VM | [Deployment](3.4-deployment.md#redeploying-after-a-code-change) |

Underlying infrastructure these scripts drive, rather than tools you invoke directly: `docker compose` (`docker-compose.yml` for production, `docker-compose.staging.yml` for staging), the shared `Dockerfile` both build from, Caddy (`Caddyfile`/`Caddyfile.staging`) for reverse-proxying and TLS, and SQLite for persistence — see [Container Deployment](3.4-deployment.md#container-deployment) for the request-flow picture and [4.1 Configuration](4.1-configuration.md#infrastructure-configuration) for exactly how each is configured (journal mode, pool size, Caddy's routing/caching rules, Docker's stage/image layout).
