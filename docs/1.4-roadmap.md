# Roadmap

The project began as a command-line prototype (`first-try`/`second-try`) implementing core Scrabble mechanics — board, tile bag, turns, word validation — and a simple game engine. It then moved to a Dioxus-based UI, before being rebuilt around a server-owned, pluggable-engine multiplayer platform. The steps below track that progression from prototype through to where the project is headed next.

## CLI Prototype ✅

Two Rust prototype crates, `first-try` and `second-try`, both centered on a terminal-based Scrabble game loop — board representation, tile bag/rack management, player turns, word lookup and validation, and basic game flow for human and computer players. The two crates reflect iteration rather than separate products; they share most of the same domain logic.

## UI Direction ✅

The `ui` crate introduced a Dioxus-based application, shifting the project toward a web and desktop presentation layer: a browser-first experience, a desktop launch path, a reusable UI surface for different clients, and a clearer separation between UI and game logic.

## MVP ✅

This is where the project stopped being a single game loop and became a server-owned, multi-client platform with pluggable engines — an Axum server holding authoritative game state, a shared pure-Rust rules crate for legality and scoring, a `GameEngine` trait (GreedyEngine is the reference implementation), and both web and desktop clients against the same API.

- Server-owned game state and rule enforcement ✅
- Human-vs-human, human-vs-engine, and engine-vs-engine games ✅ (engine-vs-engine has explicit test coverage, not just untested-but-assumed-working)
- One stable engine interface for plug-ins ✅
- Shared pure rules library used by server, clients, and engine proxies for legality and score previews ✅
- Game creation, join, start, resign, pass, exchange tiles, and move history ✅
- Server-side scoring and legality checks ✅
- Web client first ✅ (desktop client also built, ahead of this list)
- Basic reconnect support ✅ (GET /games/{id} reloads full state; session restore on client relaunch via "stay logged in")
- Deterministic tests for move legality, scoring, and endgame flow ✅ — 248 tests workspace-wide, see `IMPLEMENTATION_STATUS.md`

See `IMPLEMENTATION_STATUS.md` for the detailed, current breakdown of what's done versus what's still open.

## v1

Player identity was the first gap after the MVP: anyone with a game's URL could act as any seat, with no real notion of "this request is genuinely from the player who owns this seat." This added real player accounts (register/login with argon2-hashed passwords, unique display names), bearer-token sessions, "remember me"/"stay logged in", and seat-ownership enforcement — additive, not a breaking change to anonymous play.

- Multiple engine implementations with selectable seat assignment (only GreedyEngine exists so far)
- Engine benchmarking and head-to-head runs
- CLI and desktop clients using the same API (desktop ✅, CLI not built)
- Player identity separate from client device/session identity ✅ — register/login/validate/change-password/update-details/forgot-password, seat-ownership enforcement on every action-capable endpoint (not just `submit_action`); see `2.5-authentication.md` for what's still partial (only email verification)
- Full invitation and roster-management model ✅ — named, open, and email invitations at game creation; creator can add/remove/withdraw/resend a seat and reorder turn order after the game already exists, force-resign an unresponsive opponent mid-game, and abort (cancel) the whole game outright; see `2.7-authentication-and-invitations.md`
- Lobby flow for private and public games — still not built; today's model is invitation-based (named/open/email) rather than a browsable public lobby
- Save/load game state (save works; load requires knowing the game id, no discovery UI beyond the games list)
- Spectator mode
- Audit log of moves and server decisions
- Versioned engine contract (version strings exist, no migration path tested)
