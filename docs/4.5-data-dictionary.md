# Data Dictionary

One place to see where each game field lives across the four representations —
the authoritative `snapshot_json` blob, the denormalized DB columns, and the API
DTOs — and, crucially, *which kind* of representation each is. Ties together
[4.2 Database Schema](4.2-database-schema.md), [4.3 API Schema](4.3-api-schema.md),
and [4.4 snapshot_json Schema](4.4-snapshot-json-schema.md).

**Two Rust type layers, not three.** The domain/runtime structs (`GameSession`,
`ParticipantState`, `MoveRecord`, `ChatMessageRecord` in `game_state.rs`;
`VariantRules`/`Rack` in `rules-shared`) are what `snapshot_json` serializes —
`PersistedGame`/`PersistedVariantRules` are thin persisted mirrors, and the
nested types *are* the domain structs. So the "Rust domain / `snapshot_json`"
column below is one thing. The **DTOs** (`api` crate) are a genuinely separate
layer — the wire projection. A few enums (`SeatKind`, `GameStatus`,
`DirectionDto`, `PremiumDto`) live in `api` and are shared by both.

**Kind legend:**

- **authoritative** — the source of truth (the blob itself).
- **projection** — a denormalized copy written from the blob into a queryable column on every `save_game`.
- **blob-only** — lives only inside `snapshot_json`; no column. Reached by loading the blob, or handed straight to a DTO.
- **column-only** — a DB column with no blob field (a save-time timestamp, or a control flag).
- **computed** — derived on read (a SQL aggregate or a per-request calculation), not stored as-is.
- **separate table** — lives in its own table, looked up/joined for the DTO.

## Game

| Concept | Rust domain / `snapshot_json` | DB column | DTO (`api`) | Kind |
|---|---|---|---|---|
| game id | `PersistedGame.id` | `games.id` | `GameStateDto.id`, `GameSummaryDto.id` | projection |
| status | `PersistedGame.status` (`GameStatus`) | `games.status` | `GameStateDto.status`, `GameSummaryDto.status` | projection |
| variant / language / layout | `PersistedGame.{variant,language,board_layout}` | `games.{variant,language,board_layout}` | `GameStateDto.{variant,language,board_layout}` | projection |
| per-game ruleset | `PersistedGame.rules` (`PersistedVariantRules`) | — | — (effects surface via `board`/scores) | blob-only |
| turn number | `PersistedGame.turn_number` | `games.turn_number` | `GameStateDto.turn_number` | projection |
| current seat | `PersistedGame.current_seat` | `games.current_seat` | `GameStateDto.current_seat`, `GameSummaryDto.current_seat` | projection |
| winner seat | `PersistedGame.winner_seat` | `games.winner_seat` | `GameStateDto.winner_seat` | projection |
| end-of-game bonus | `PersistedGame.final_bonus_{seat,points}` | — | `GameStateDto.final_bonus_{seat,points}` | blob-only |
| creator | `PersistedGame.creator_player_id` | — | `GameStateDto.creator_player_id` | blob-only |
| RNG seed | `PersistedGame.random_seed` | `games.random_seed` | — | projection (internal) |
| board | `PersistedGame.board` (`Vec<BoardCellDto>`) | — | `GameStateDto.board` | blob-only |

| move-time limit | `PersistedGame.move_time_limit_seconds` | — | `GameStateDto`/`GameSummaryDto.move_time_limit_seconds` | blob-only |

| stats-settled marker | — | `games.stats_settled_at` | — | column-only (single-fire rating gate) |
| **the snapshot itself** | *(the whole `PersistedGame`)* | `games.snapshot_json` | — | **authoritative** |

## Cross-cutting / computed (no single home)

| Concept | Source | DTO (`api`) | Kind |
|---|---|---|---|
| last activity | `max(game_moves.created_at)` ⟵ `games.created_at` fallback | `GameSummaryDto.last_activity_at` | computed (SQL aggregate) |
| last message time | last `ChatMessageRecord.created_at` in the blob | `GameSummaryDto.last_message_at` | computed from blob |
| bag count | `len(PersistedGame.bag)` | `GameStateDto.bag_count` | computed from blob |
| viewer relationship | per-request, from who's asking vs seats/creator | `GameSummaryDto.relationship` | computed per-request |
| unread-chat flag | client compares `last_message_at` vs its stored watermark | *(client-side)* | computed client-side |

## Player / session (outside the game aggregate)

These have their own tables and DTOs, not the `snapshot_json` blob:

- **player** — `players` table → `PlayerDto` (`id`, `display_name`, `email`, `created_at`, `last_seen_at`).
- **session** — `sessions` table (`token_hash`, `created_at`, `last_seen_at`, `expires_at`, `stay_logged_in`) → `PlayerSessionDto` on login (token handed back once). See [2.5 Authentication](2.5-authentication.md) and the sessions notes in [4.2](4.2-database-schema.md#sessions).
- **rating** — `player_ratings` (current) / `rating_history` (per-game) → surfaced on `ParticipantDto` (above) and the Stats view.

---

**Client-local state (not a schema layer here):** the web/desktop client also
persists two small structs on-device (`StoredAuth`, `StoredChatWatermarks`) —
client preferences, not game data, so they're out of the tables above.
Documented separately in [4.6 Client-Local Storage](4.6-client-local-storage.md).
