# Configuration

Environment and application configuration — the stuff that changes rarely
and you look up rather than read start to end. For system topology and the
deployment diagram, see [1.1 Architecture](1.1-architecture.md)'s "System
Overview". For one-time provisioning, see [Setup](3.1-setup.md). For
day-to-day commands, see the lifecycle docs (linked from
[docs/README.md](README.md)) and [Tools](3.0-tools.md). For what's
persisted, see [4.2 Database Schema](4.2-database-schema.md) and
[4.3 API Schema](4.3-api-schema.md).

## Environments

Three genuinely different places this project runs, easy to conflate since some of the same commands (`docker compose ...`) work in all three:

### Local dev machine

Where you write code, run tests, and build the images that get deployed. Has the full source tree; nothing here is what's actually serving live traffic.

**Components**: Rust toolchain + workspace crates, Docker Engine (used here only to *build* images and optionally run the stack or the staging environment for local testing — not to serve real traffic), the git clone itself (pushed to/pulled from GitHub).

**Directory structure** (repo root, `~/tile-lite-elite` in this WSL setup):

```
tile-lite-elite/
├── crates/                  # the six workspace crates
│   ├── api/                 # shared request/response DTOs
│   ├── rules-shared/        # pure rules/scoring/move-generation
│   ├── engine-core/         # GameEngine trait + GreedyEngine
│   ├── server-game/         # Axum backend
│   ├── ui/                  # Dioxus web/desktop client
│   └── admin-cli/           # tile-lite-elite-admin operator CLI
├── old-crates/              # early prototypes, kept for design precedent only
├── docs/                    # this file and other design docs
├── scripts/                 # admin.sh, deploy.sh, deploy-staging.sh, services.sh, setup-dev-environment.sh, desktop.sh
├── data/                    # local dev SQLite file (TILE_LITE_ELITE_DATABASE_URL's default)
├── target/                  # cargo build output (gitignored)
├── .cargo/config.toml       # sccache + wasm rustflags (see 3.1-setup.md's Troubleshooting)
├── Dockerfile, docker-compose.yml, docker-compose.staging.yml, Caddyfile, Caddyfile.staging, .dockerignore
└── Cargo.toml                # workspace manifest
```

### Staging (local)

A second, disposable deployment target that runs *on the dev machine* — not a separate physical environment, but different enough in shape (its own containers, port, volume) to treat as a peer of the other two rather than a footnote. Exists purely to test a deployment (especially a schema migration) before it reaches production. See [Testing, CI & Release](3.3-testing-ci-and-release.md#staging-environment) for the full workflow.

**Components**: same two containers as production (`server`, `web`/Caddy), built from the identical `Dockerfile`, run via the separate `docker-compose.staging.yml` + `Caddyfile.staging` (plain HTTP, no domain, no TLS — see [Setup](3.1-setup.md#https) for why production needs a real hostname and staging doesn't).

**Directory structure**: none — it's ephemeral, built from whichever git ref you point it at (usually the working tree, or a specific commit via `at <ref>`/`at prod`) and torn down with `down`/`reset`. The only persistent thing is its data volume:

```
tile-lite-elite-staging-server-1, tile-lite-elite-staging-web-1   # containers
tile-lite-elite-staging-data                                       # volume — /data in the server container
```

### Oracle Cloud VM (production)

Where the live deployment actually runs. **Does not have the source tree at all** — no git clone, no `crates/`, nothing to build. Just the compose file and whatever Docker itself stores (images, volumes) — `scripts/deploy.sh` builds everything locally and ships only the finished images plus `docker-compose.yml` over.

**Components**: Docker Engine only. Two running containers (`tile-lite-elite-server-1`, `tile-lite-elite-web-1`) and three named volumes (`tile-lite-elite-data`, `tile-lite-elite-caddy-data`, `tile-lite-elite-caddy-config`) — see [Container Deployment](3.4-deployment.md#container-deployment) for what each holds.

**Directory structure** (`~/tile-lite-elite` on the VM — same path as local, different machine, don't let that imply it's the same *kind* of directory):

```
~/tile-lite-elite/
└── docker-compose.yml       # the only file that lives here
```

That's genuinely everything on disk outside of Docker's own storage. What's *inside* the containers (via `docker compose exec <service> ls ...`):

```
server container:
├── /usr/local/bin/
│   ├── server-game          # the release binary actually running
│   └── tile-lite-elite-admin       # copied in but not running — invoked on demand via `docker compose exec`
└── /data/                   # the tile-lite-elite-data volume
    └── tile-lite-elite.sqlite3

web container:
├── /srv/                    # the built web client (index.html, assets/, wasm/) — served as static files
├── /etc/caddy/Caddyfile     # baked into the image at build time, not a volume
├── /data/caddy/             # the caddy-data volume — obtained TLS certificates live here
└── /config/caddy/           # the caddy-config volume — Caddy's own runtime state
```

Staging's server container has the same `/usr/local/bin`/`/data` shape; its web container has no `/data/caddy`/`/config/caddy` (no TLS to persist).

## Environment Variables

| Environment variable | Default | Description |
|---|---|---|
| `TILE_LITE_ELITE_BIND` | `127.0.0.1:3000` | Server listen address |
| `TILE_LITE_ELITE_DATABASE_URL` | `sqlite://data/tile-lite-elite.sqlite3` | SQLite database path |
| `TILE_LITE_ELITE_API_BASE_URL` | `http://127.0.0.1:3000` | Backend URL used by clients. Set at *build* time (`option_env!`), not runtime. An explicit empty string means "same origin as the page" — see [Container Deployment](3.4-deployment.md#container-deployment) |
| `TILE_LITE_ELITE_UI_PORT` | `8080` | Web dev server port |
| `TILE_LITE_ELITE_PUBLIC_BASE_URL` | `http://127.0.0.1:8080` | Where the web client is served from — used only to build the link inside emails (password reset, welcome, invitation) |
| `RESEND_API_KEY` | unset | [Resend](https://resend.com) API key for sending email. Unset means no provider is configured — every email-triggering flow (register, invite, forgot-password) still works, it just logs the message instead of sending it (see `crates/server-game/src/email.rs`). Email wording itself lives in `crates/server-game/emails/*.txt`, not Rust code. Set via a `.env` file next to `docker-compose.yml`, **not** committed to git — see [Container Deployment](3.4-deployment.md#container-deployment) |
| `RESEND_FROM_ADDRESS` | `Tile Lite Elite <noreply@mail.tileliteelite.com>` | The `From:` address emails are sent as. Deliberately a subdomain (`mail.tileliteelite.com`), not the root domain — keeps Resend's required SPF/DKIM records completely separate from the root domain's own mailbox (GoDaddy Email) records, so nothing here can affect real mail deliverability |
| `RUST_LOG` | `server_game=info,tower_http=info,warn` | Log verbosity for `server-game`. See [Logging](3.5-production-support.md#logging) |
| `TILE_LITE_ELITE_BUILD_ID` | unset | Optional build identifier baked in at *build* time (`option_env!`), appended as SemVer build metadata to the app version (e.g. `0.4.6+a1c9f02`). Unset (the default when building by hand) shows only `Major.Minor.Patch`; `scripts/deploy.sh`/`deploy-staging.sh` set it automatically. See [Versioning](#versioning) |
| `PROD_URL` | `https://tileliteelite.com` | Used only by `scripts/deploy-staging.sh`'s `at prod`/`verify` modes, to know where to check production's live version. See [Staging Environment](3.3-testing-ci-and-release.md#staging-environment) |
| `STAGING_URL` | `http://localhost:8081` | Used only by `scripts/deploy.sh`, to confirm local staging is running the exact commit about to be shipped before proceeding. See [Redeploying](3.4-deployment.md#redeploying-after-a-code-change) |

### Environment-specific values

The table above gives each variable's bare/local-dev default; here's what actually differs across the three environments day to day (blank = not applicable/not set there):

| Variable | Local dev | Staging | Production |
|---|---|---|---|
| `TILE_LITE_ELITE_BIND` | `127.0.0.1:3000` | `0.0.0.0:3000` (in-container; only reachable via `web`) | `0.0.0.0:3000` (in-container; only reachable via `web`) |
| `TILE_LITE_ELITE_DATABASE_URL` | `sqlite://data/tile-lite-elite.sqlite3` (bind-mounted file) | `sqlite:///data/tile-lite-elite.sqlite3` (`tile-lite-elite-staging-data` volume) | `sqlite:///data/tile-lite-elite.sqlite3` (`tile-lite-elite-data` volume) |
| `TILE_LITE_ELITE_PUBLIC_BASE_URL` | `http://127.0.0.1:8080` | `http://localhost:8081` | `https://tileliteelite.com` |
| `TILE_LITE_ELITE_API_BASE_URL` (build-time) | `http://127.0.0.1:3000` | `""` (same-origin) | `""` (same-origin) |
| Web client port | `8080` (`dx serve`) | `8081` (Caddy, plain HTTP) | `80`/`443` (Caddy, HTTPS) |
| `RESEND_API_KEY` | unset (emails log instead of send) | unset unless deliberately set | set via `.env` on the VM |

The dev/staging/production split for `TILE_LITE_ELITE_BIND` and same-origin `TILE_LITE_ELITE_API_BASE_URL` is what lets one compiled wasm bundle work unmodified in both staging and production — see [Container Deployment](3.4-deployment.md#container-deployment)'s "Why one image serves both" note.

## Infrastructure Configuration

The actual runtime settings of the two pieces of infrastructure this project depends on, verified against the live config/code rather than assumed — worth checking here before assuming a default.

### SQLite

`persistence::connect` (`crates/server-game/src/persistence.rs`) does exactly this and nothing else:

```rust
SqliteConnectOptions::from_str(database_url)?.create_if_missing(true)
```

`persistence::connect` sets `journal_mode` explicitly (`0.3.0`, see below); synchronous and foreign-key pragmas are left at SQLite's own defaults. Verified directly against a live database (`sqlite3 <file> "PRAGMA journal_mode; PRAGMA synchronous; PRAGMA foreign_keys;"`):

| Pragma | Actual value | Means |
|---|---|---|
| `journal_mode` | `wal` (since `0.3.0`; was `delete` before) | Readers no longer block behind a writer's transaction — matters once more than a handful of players are concurrently active. Sets `-wal`/`-shm` sidecar files alongside the main database file (`3.2-development.md`'s "Resetting the Database" step already correctly hedges on this — "may not exist depending on journal mode" — now they do). Volume backups (`3.5-production-support.md#backups`) already tar the whole directory, so this needed no backup-procedure change. |
| `synchronous` | `2` (`FULL`) | Unchanged by the `journal_mode` switch (verified — sqlx doesn't adjust it automatically the way the `sqlite3` CLI's shorthand modes sometimes imply). Every commit still `fsync`s before returning. `NORMAL` (`1`) is commonly paired with WAL for a latency win, since WAL's own durability model needs less aggressive fsyncing — not changed here since it wasn't part of this release's scope, but worth a deliberate look if write latency ever becomes a real concern. |
| `foreign_keys` | `0` (off) | **The `foreign key to X.id` relationships documented throughout [4.2 Database Schema](4.2-database-schema.md) are not actually enforced by SQLite** — they're honored by application logic only (verified: `delete_player`/`delete_game` in `persistence.rs` already delete child rows before parent rows inside a transaction, in the right order). Turning this pragma on wouldn't currently change that, though: `crates/server-game/migrations/0001_baseline.sql` has no `foreign key ... references ...` clauses at all — the schema doc's "foreign key to X" language describes the *logical* relationship, not a declared SQL constraint. Real enforcement needs both the pragma **and** adding actual `references` clauses, and SQLite can't `alter table` a constraint onto an existing table — it means recreating each table. That's schema-migration-sized work, not a config change; not undertaken as part of this pass. |

Connection pool: `SqlitePoolOptions::new().max_connections(5)` — one shared pool per process, same in every environment (dev/staging/production all run a single `server-game` process each; there's no multi-process fan-out that would make 5 a bottleneck).

### Caddy

Version pinned via `caddy:2-alpine` (`Dockerfile`'s `runtime-web` stage) — floats on Caddy's `2.x` line, not pinned to an exact patch version.

- **Compression**: `encode zstd gzip` — zstd preferred, gzip fallback, for every response Caddy serves (static assets and proxied API alike).
- **Routing**: an explicit `route { }` block, not Caddy's default directive ordering — needed because Caddy's automatic ordering runs `try_files`/`file_server` *before* `reverse_proxy`, which would rewrite every API path to `/index.html` before the `@api` matcher ever saw it. The `@api` matcher lists exact paths/prefixes (`/health /engines /auth/* /games /games/* /players/* /invitations/* /dictionaries/*`) — a new route added to the server's API needs a matching entry here too, or it'll silently fall through to the SPA fallback instead of reaching `server-game`. (`Caddyfile.staging` duplicates this same matcher list — see [3.3 Testing, CI & Release](3.3-testing-ci-and-release.md#staging-environment) — so a new route needs updating in both files.)
- **Caching**: `/assets/*` gets `Cache-Control: public, max-age=31536000, immutable` (content-hashed filenames, safe to cache forever); `index.html` itself is deliberately left with no caching directive, since it's what points at the current asset hashes.
- **TLS**: fully automatic via Let's Encrypt — see [3.1 Setup](3.1-setup.md#https) for the sslip.io/real-domain mechanics. No manual certificate handling anywhere.
- **Admin API**: Caddy's own runtime admin endpoint (`localhost:2019`, visible in its startup log) is **not** exposed via `docker-compose.yml`'s `ports:` — reachable only from inside the container, not the host or internet.
- **No explicit resource limits**: neither `docker-compose.yml` nor `docker-compose.staging.yml` sets `mem_limit`/`cpus`/`deploy.resources` on any service — both containers share whatever the VM (1GB RAM + swap, see [3.1 Setup](3.1-setup.md#oracle-cloud-vm-setup)) or dev machine has available, uncapped.

### Docker

Multi-stage `Dockerfile`, one shared `builder` stage (`rust:1-bookworm`) feeding two final images — see [3.4 Deployment](3.4-deployment.md#container-deployment) for the request-flow/volume picture; this is the build side.

| Stage | Base image | Produces |
|---|---|---|
| `builder` | `rust:1-bookworm` | Compiles `server-game`/`admin-cli` (release) and the web client (`dx build --release`) — not shipped itself, just feeds the two below |
| `runtime-server` | `debian:bookworm-slim` + `ca-certificates` | `server-game` + `tile-lite-elite-admin` binaries only |
| `runtime-web` | `caddy:2-alpine` | The built wasm/static bundle + `Caddyfile` |

Built images (local, `docker compose build`, `TILE_LITE_ELITE_BUILD_ID` set): `tile-lite-elite-server` ≈199MB, `tile-lite-elite-web` ≈92MB — small enough that `scripts/deploy.sh`'s `docker save | gzip` / `scp` round trip (see [3.4 Deployment](3.4-deployment.md#redeploying-after-a-code-change)) stays fast even over a modest connection. `.dockerignore` excludes `target`, `.git`, `data`, `.dioxus`, log/pid files, and `node_modules` from the build context, so none of those slow down the context upload to the Docker daemon or accidentally end up in an image layer.

Both runtime images define a `HEALTHCHECK` (since `0.3.0`): `runtime-server` curls its own `/health` directly (`curl` added to that stage solely for this — the only new package either image needed); `runtime-web` hits Caddy's admin API (`127.0.0.1:2019/config/`, alpine's built-in busybox `wget`) rather than the public `/health` path. That wasn't the first design — hitting `/health` through Caddy's own reverse-proxy path would check more (that Caddy can actually reach the server, not just that Caddy is up), but it **failed on real production once actually deployed**: Caddy's global HTTP→HTTPS auto-redirect fires on `:80` for any `Host`, including a bare loopback probe matching none of the configured hostnames, and the `:443` TLS handshake then fails (no certificate for that SNI). Staging's `Caddyfile.staging` has no TLS/redirect at all, so this only surfaced in production — worth remembering next time something "verified in staging" needs re-checking against whatever specifically differs in production, not treated as fully proven. (Also hit along the way: busybox `wget` resolves `localhost` to the IPv6 loopback first, and Caddy's admin listener is IPv4-only — use `127.0.0.1` explicitly or get a misleading "connection refused.") Neither compose file overrides the `--interval`/`--timeout`/`--start-period` set in the `Dockerfile`. Visible via `docker compose ps` (a `healthy`/`unhealthy`/`starting` column) or `docker inspect --format='{{.State.Health.Status}}' <container>`.

## Versioning

Two independent version numbers, on purpose — see the doc comments on
`api::API_VERSION` and each binary's `app_version()` (in `server-game`'s
and `tile-lite-elite-ui`'s `main.rs`) for the full rationale.

**API contract version** (`Major.Minor`, e.g. `1.1`) — lives once in the
shared `api` crate as `api::API_VERSION`, so both the server and any given
client binary embed whatever it was at *their own* build time. Bump
`major` for a breaking DTO/route change, `minor` for an additive,
non-breaking one. The desktop client checks this against the server's
`/health` response the moment it first connects
(`check_api_version`/`compare_api_version` in `crates/ui/src/app.rs`): a
major mismatch blocks further use with an "update the app" message, a
minor mismatch shows a soft, non-blocking notice. There's deliberately no
patch/build component here — those never change the wire contract, so
including them would make the check fire on every routine bugfix deploy.
The only production client today is the wasm build shipped in the same
image as the server, so it's always in step and a mismatch has never
actually been observed — that won't stay true forever (e.g. a desktop
build lagging a server redeploy), so bump this on every route/DTO change
as a habit now, not just once it starts mattering.

**Bump per real wire change, even when production skips the intermediate
versions.** The version number describes the *artifact*, not production's
current view of it — staging or a test box can be checked out at any
committed version, so each wire change earns its own increment to keep a
cross-environment `api_version` comparison meaningful. Don't collapse an
intermediate bump just because prod jumped past it. Worked example: prod
sat at `1.4` while dev moved `1.4 → 2.0` (timestamps became integers —
breaking) `→ 2.1` (added the `aborted` `GameStatus` variant — additive)
across two undeployed commits; both bumps were kept, and the next deploy
simply takes production `1.4 → 2.1` in one step.

**App/build version** (`Major.Minor.Patch[+build]`) — `Major.Minor.Patch`
is **workspace-inherited**: it lives once in the root `Cargo.toml`'s
`[workspace.package]` (`version = "…"`, currently `0.4.6`) and every crate
takes it via `version.workspace = true`, so a bump is a single one-line
edit (converted from per-crate versions 2026-07-21). The owner bumps the
dev tree to the next patch **right after each production deploy**, so the
working tree always leads production by one version — patch for fixes,
minor for new functionality. An optional
`+<build>` suffix (standard SemVer build metadata) is appended when
`TILE_LITE_ELITE_BUILD_ID` is set at compile time — e.g. a git short SHA,
for telling builds apart:

```bash
# Internal/test build with a build id
TILE_LITE_ELITE_BUILD_ID=$(git rev-parse --short HEAD) cargo build -p server-game --release

# Production release — no build id set, shows "0.4.6" not "0.4.6+..."
cargo build -p server-game --release
```

`scripts/deploy.sh` and `scripts/deploy-staging.sh` both set
`TILE_LITE_ELITE_BUILD_ID` automatically from `git rev-parse --short HEAD`
and pass it through `docker-compose.yml`'s `build.args` into the
`Dockerfile` — every image built through either script carries its commit
without a manual step; building by hand just omits the suffix.

`server-game` logs its app version alongside the API version at startup,
and serves it in the `app_version` field of `GET /health`'s response body
(`api::HealthDto`); the desktop client puts its app version in the window
title. `/health` specifically is what [Staging
Environment](3.3-testing-ci-and-release.md#staging-environment)'s `deploy-staging.sh
at prod`/`verify` read to find out which commit is actually live, without
needing to SSH in.
