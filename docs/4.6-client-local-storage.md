# Client-Local Storage

Small bits of state the web/desktop client keeps **on the device**, across app
restarts — client preferences, not game data (which is server-owned; see
[4.5 Data Dictionary](4.5-data-dictionary.md)). Derived from
`crates/ui/src/local_storage.rs`.

**Where it's kept differs by platform** (there's no browser `localStorage` on
desktop):

- **Web** — browser `localStorage`, under fixed keys.
- **Desktop** — a small JSON file under the OS config directory
  (`dirs::config_dir()/tile-lite-elite/…`).

Both are best-effort: a failed read falls back to the type's `Default`, and a
failed write is ignored — losing this state only costs a re-login or a reset
unread indicator, never anything the server cares about.

## `StoredAuth`

| Field | Type | Notes |
|---|---|---|
| `remembered_name` | string \| null | Pre-fills the display-name field next time. Pure convenience, no security weight — kept when "Remember me" is checked. |
| `session_token` | string \| null | The actual bearer token. Persisted **only** if the user checked "Stay logged in" — this is the client-side half of what that checkbox means (the server no longer ties expiry to it; see [2.5 Authentication](2.5-authentication.md) and the sessions notes in [4.2](4.2-database-schema.md#sessions)). On restart the client re-validates it via `POST /auth/validate` and drops it if it's no longer valid. |

- Web key: `tile_lite_elite_auth`
- Desktop file: `…/tile-lite-elite/auth.json`

**Security note:** persisting a bearer token in `localStorage`/a plaintext file
is the usual convenience-vs-exposure trade-off, and is why it's opt-in ("Stay
logged in") rather than the default. Without it, closing the browser drops the
token and the next visit requires a fresh login. The token's exposure window is
bounded by the session's absolute + idle limits regardless (see
[4.2](4.2-database-schema.md#sessions)).

## `StoredChatWatermarks`

| Field | Type | Notes |
|---|---|---|
| `last_seen` | map<game_id, i64> | Per game, the `created_at` (unix seconds) of the last chat message this device has seen. A game whose latest message time doesn't match its entry (or has no entry) shows as having unread chat. |

- Web key: `tile_lite_elite_chat_seen`
- Desktop file: `…/tile-lite-elite/chat_watermarks.json`

Purely local: there is **no server-side read-receipt concept**, so unread state
is per device/browser and never shared or synced.
